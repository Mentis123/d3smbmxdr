Power Automate Email Agent

Personalised Outreach Drafts
with human approval

Standalone guide for building a practical flow that scans your Outlook “Customer” folders, extracts first name + email, creates personalised draft emails with your supplied content, and lets you test on yourself before wider rollout.

1) Use Case Definition

Goal: generate draft emails (not auto-send) so you can review each one before release.

Input

Outlook “Customer” folder + sub-folders, customer contact card metadata, campaign template body and subject rules.

Output

Personalised draft in Outlook with first-name intro and customised subject line.

Control

Human-in-the-loop approval gate and test-mode switch for self-only validation.

2) Recommended Flow Architecture

Trigger: Manual / Scheduled Connector: Office 365 Outlook Optional: AI Builder / Copilot prompt
  1. Load approved campaign content + variables (offer, CTA, signature).
  2. List messages/contacts under “Customer” root and iterate sub-folders. Note to Codex: investigate possibility of using Dataverse contact sync via Exchange server-side sync to use Contacts as a means to discover email recipients instead, which may be more reliable than inspecting Outlook folders.
  3. Extract recipient first name + email (fallback to parsing display name).
  4. Compose subject and body from template with token replacement.
  5. If test mode = true, rewrite recipient to your own email address.
  6. Create draft message in Outlook, tag category “Agent-Draft-Pending”.
  7. Send approval summary to you (Teams or email) with draft count.

3) Data Rules & Personalisation Logic

First name logic

Use givenName when available. Else split displayName by spaces and take token 1. If blank, fallback to “Hi there”.

Subject logic

Pattern: “<FirstName>, quick note on <topic>”. Keep under 65 chars where possible.

Body logic

Template blocks: greeting, 2–3 lines of value, CTA, sign-off. Keep plain-English tone.

Dedup logic

Do not draft more than one email per recipient per campaign run; hash recipient+campaign ID.

4) Safety, Governance, and Testing

Test-first controls

  • Boolean variable test_mode defaults to true in dev and UAT.
  • When true, all drafts route to your own inbox with original recipient in body metadata.
  • Approval required before any run with test_mode=false.

Operational controls

  • Cap run volume (example: 25 drafts/run) to reduce accidental blast radius.
  • Store run log (recipient, timestamp, campaign id, draft id, operator).
  • Add kill switch variable to halt processing immediately.

5) 30-Minute Working Session Agenda

  1. Confirm mailbox/folder shape and contact data quality (5 min).
  2. Build minimum viable flow live (10 min).
  3. Add personalisation + test mode + approval branch (10 min).
  4. Run live test to self, review drafts, define next hardening actions (5 min).

Outcome: working, review-first drafting agent you can iterate safely.
